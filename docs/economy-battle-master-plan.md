## 0. ภาพรวม — ทุกอย่างเป็นวงเดียวกัน

```
เรียน (quiz/SRS) ──daily quest──> ปลดตัวคูณรายได้ ×1.25
                                        │
ฟาร์ม/บ้าน (idle, capped) ──────────────┤──> เหรียญ (สกุลเดียว)
                                        │         │
ไต่หอคอย (RPG) ──ปลดรายได้ประจำวัน──────┘         │ ใช้
                    ▲                              ▼
                    │ ใช้พลังเพ็ท            กาชา (Genshin-style)
                    │                              │
              เพ็ท (rarity+grade+ธาตุ) <──dupe→+เกรด──┘
```

หลักการเดียวที่คุมทุกอย่าง: **เศรษฐกิจสกุลเดียว → ต้องคุมก๊อกน้ำ (faucet) ให้แน่น** เพราะไม่มีสกุลที่ 2 ช่วยกันเฟ้อ ทุกระบบที่ "จ่ายเหรียญ" ต้องมี cap หรือเป็นตัวคูณ ไม่ใช่เหรียญดิบไม่อั้น

## 1. เศรษฐกิจ (สกุลเดียว)

### ก๊อกน้ำ (faucets) — คุมให้แคบ
| แหล่ง | ปัจจุบัน | เป้าหลังปรับ |
|---|---|---|
| บ้าน (idle) | Lv2 600 / Lv6 5k / Lv12 55k /วัน | คงไว้ (เป็นแกน) |
| ฟาร์มพืชยาว | เต็มแปลง 15k–220k ⚠️ | **ลด sellprice 50–60%** → ฟาร์มเต็ม ≈ 1–1.5× รายได้บ้าน |
| Quiz | cap 300/วัน | คงไว้ |
| Study | cap 150/วัน | คงไว้ — แต่คุณค่าหลักคือ "ตัวคูณ" ไม่ใช่เหรียญดิบ |
| หอคอย (ใหม่) | – | bonus รายวันสูงสุด ≈ เทียบบ้านท็อป **ไม่ใช่ทวีคูณ** |

### ตัวคูณจากการเรียน (หัวใจ — ทำให้เรียน = ศูนย์กลาง)
- **Daily quest**: ทบทวน SRS ครบโควต้า + quiz ≥10 ข้อ → **รายได้ idle ×1.25 เป็นเวลา 24 ชม.**
- ไม่ใช่เหรียญดิบ → ไม่ inflate · คนอยากได้เพ็ทเร็วต้องเรียนเพื่อเปิดตัวคูณ
- (ออปชัน) streak ต่อเนื่อง → ตัวคูณแรงขึ้นเล็กน้อย เช่น 5 วัน → ×1.35

### อ่างระบาย (sinks) — ทำให้เหรียญมีค่า
บ้าน (มี, ถึง 1.42M ✅) · grade-up เพ็ท (copies + เหรียญ) · **กาชา** · ของตกแต่ง/คอสเมติก (อนาคต)

### กฎเหล็กตอนเพิ่มของใหม่
ทุกครั้งที่เพิ่มมินิเกม/ฟีเจอร์ที่จ่ายเหรียญ → **ต้องใส่ cap หรือทำเป็นตัวคูณ** ไม่งั้นสกุลเดียวเฟ้อทันที

## 2. กาชา (Genshin-style, สกุลเดียว)

| รายการ | หมุด (อัปเดต — ใจป้ำ) |
|---|---|
| สกุลที่ใช้ | เหรียญ (สกุลเดียว) |
| Single pull | **1,000 🪙** |
| 10-pull | **10,000 🪙 → ได้ 11 ตัว** ("เปิด 10 แถม 1" ≈ ลด 9%) |
| Soft pity | ~75 pulls (rate legendary ไต่ขึ้น) |
| Hard pity | 100 pulls การันตี legendary |
| Featured | 50/50 → พลาดครั้งนี้ การันตีครั้งหน้า |

**ตั้งใจให้ "ถูก/ใจป้ำ"** (เกมชั้นปี ไม่มีเงินจริง — คนได้เก็บเยอะ = สนุก) แต่ **1,000 ถูกมากเทียบรายได้** (Lv6 ~5 จุ่ม/วัน, Lv12 ~55 จุ่ม/วัน) → **ย้าย "การลุ้น" ออกจากราคา ไปไว้ที่:**
- **chase = ตัว featured + pity ต่อ banner (50/50)** — จุ่มถูกแต่จะได้ "ตัวที่อยากได้" ยังต้องลุ้น
- **sink ระยะยาว = dupe→เกรด** (แม็กซ์ทีมหลักกิน copies เยอะ = ที่เทเหรียญจริง)
- **ต้อง cap ฟาร์มให้อยู่** (§1) ไม่งั้นรายได้พุ่งจนจุ่มฟรีวันแรก
- ⚠️ ที่ราคานี้ hard pity 100 = 100k → Lv12 ~2 วัน/การันตี legendary · ถ้าอยากให้ปลายเกมไม่ง่ายไป ค่อยดัน pity ขึ้น (ปล่อยใจป้ำก็ได้)

**Dupe = constellation:** สุ่มได้ตัวซ้ำ → **+1 เกรด** (ไม่ใช่ทิ้ง) ผูกกาชาเข้ากับระบบเกรดเพ็ทโดยตรง

## 3. เพ็ท — รื้อสมดุล

### ปัญหาที่เจอ (วินิจฉัยแล้ว)
- เกรด I–XII = ต้อง 24 copies/ตัวเพื่อแม็กซ์ → grind ยาวเกินหนึ่งเทอม
- Potential affix 7 สเตตัส แต่ 6/7 ใช้แค่ตอนสู้ (ยังไม่มา) → รู้สึกเยอะ-เปล่าประโยชน์
- ธาตุ/สเตตัสสู้ ยังไม่มีที่ใช้ (รอ battle/หอคอย)
- เพ็ทไม่มี payoff ชัด → เก็บมาแล้วแค่ออกเหรียญนิดหน่อย

### หลักแก้: จูนเทียบหอคอย ไม่ใช่จูนเดี่ยว
**อย่าตั้งตัวเลขเพ็ทลอยๆ** — ตั้ง "หอคอยกี่ชั้น + เป้าพลัง/เทอม" ก่อน แล้วถอยมาตั้งเกรด/copies/กาชา

### หมุดเริ่มต้น
- **เกรด:** เก็บแลดเดอร์ไว้ได้ แต่ตั้งเป้าให้ **ผ่านหอคอยได้ที่เกรดกลาง ~5–6** ด้วยทีม 4 ตัว · เกรด 7–12 = "ของอวด/PvP" ไม่ใช่เงื่อนไขผ่านด่าน (แก้ที่ framing ไม่ต้องลบ)
  - ถ้าอยากลดจริง: ตัดเหลือ ~I–VIII, copies รวม ~12–14
- **Potential:** เปิดเป็นเลเยอร์ตาม battle — v1 เปิดแค่ affix ที่ engine อ่านจริง (ATK/HP/Crit) · `critDmg/lifesteal/dodge` เปิดตอน engine รองรับ · `dailyCoins` ใช้ได้เลย (income)
- **ธาตุ:** เก็บใน data โชว์เฉพาะในหน้า/ระบบสู้
- **จำนวนเพ็ท:** curate 42 → **27 ตัว** (ดู §3.5 Roster Rework)
- **จุดประสงค์เพ็ทแต่ละเฟส:** ก่อน battle = รายได้ + คอลเลกชันโชว์ · หลัง battle = ทีมไต่หอ/PvP

## 4. หอคอย (RPG climb) — payoff ของเพ็ท + ปลดรายได้

- **โมเดล:** ไต่ชั้น PvE ด้วยทีมเพ็ท → auto-resolve (ใช้ battle engine)
- **ปลดรายได้:** `towerBest` (ชั้นสูงสุด) → bonus เหรียญรายวันแบบ flat (ยิ่งสูง ยิ่งได้) ผ่าน `getTowerBonus(floor)` ใน `useDaily.js`
- **หมุดเริ่มต้น:**
  - หอคอย v1 = ~50 ชั้น
  - เป้า/เทอม: เล่นสม่ำเสมอ → ถึง ~ชั้น 30–40
  - ชั้น 30 ผ่านได้ด้วยทีม 4 ตัว เกรดเฉลี่ย ~5 ธาตุผสม
  - bonus ladder (จาก roadmap): 10–19 → +500, 30–49 → +4,000, 75+ → +20,000 /วัน
- **faucet note:** bonus หอคอย = รายได้ → นับรวมในแผน cap (ข้อ 1) ตั้งให้ท็อป ≈ บ้านท็อป ไม่ใช่หลายเท่า

## 5. Battle System — โมเดล + ผลบาลานซ์ (sim แล้ว 16 มิ.ย.)

> โมเดลเคาะแล้ว + รัน balance sim แล้ว — ดูบล็อก "ตัดสินใจแล้ว / ผลบาลานซ์ / Passive / อนิเมชัน" ท้ายหัวข้อ

**โมเดล:** async client-side auto-resolve — ไม่ใช่ realtime (สแตก client-only ทำ realtime ไม่ได้) · ทีม 3–4 ตัว

**Engine (`utils/battleEngine.js`) — แกน v1:**
- พลังมาจาก **rarity (ฐาน) + เกรด (ตัวคูณ) + ธาตุ (matchup)** — 3 แกน อ่านง่าย
- Damage ≈ ATK × elementMult × critRoll · มี HP pool · ทีมไหนเลือดหมดก่อนแพ้
- ธาตุ ✊✌️✋: ได้เปรียบ **×1.25** / เสียเปรียบ **×0.85** (จูนจาก sim แล้ว — เดิม 1.5/0.67 ชนะขาด)

**โหมด:**
- **Tower (PvE)** = โหมดหลัก ทำก่อน
- **PvP async** = snapshot ทีมคู่ต่อสู้เก็บใน Firestore → ดึงมา resolve ฝั่ง client (เฟสหลัง)

**Data ที่ต้องมี:** `data/towerFloors.js` (ทีมบอทต่อชั้น) · stats เดิมใน userSchema

### ✅ ตัดสินใจแล้ว (จาก sim)
- **โมเดล = Team brawl** — 4v4 บนสนามพร้อมกัน, เล่นเป็นรอบ, แต่ละรอบทุกตัวที่ยังไม่ตายได้ตี 1 ที **ลำดับสุ่ม** (กัน first-mover → แมตช์กระจก ~50% ✓), ตายทีละตัว (ไม่ใช่ tag 1v1)
- **ลำดับการตี = สุ่ม** ไม่ต้องมี stat speed · **เป้า = random target** (กระจาย ลด snowball)
- ทีม **4 ตัว** แถวเดียว (ยังไม่มีหน้า-หลัง) · **ไม่มี energy** · หอคอยรีทรายฟรี
- จบเมื่อฝั่งหนึ่งตายเกลี้ยง หรือครบ 30 รอบ → เลือดรวม %มากกว่าชนะ
- แสดงผล = **compute-then-replay** (deterministic+seeded) → animate log + **ปุ่มเร่ง ×2** (ไม่มี skip เต็ม) · auto-fast เฉพาะฟาร์มชั้นซ้ำ

### 📊 ผลบาลานซ์ (`scripts/battle-sim.mjs`)
- ✅ แมตช์กระจก ~50% (แก้ first-mover ด้วยลำดับสุ่ม)
- ⚠️ **combat snowball แรง** — ได้เปรียบนิดเดียวลามชนะขาด: เกรด +1 ~81%, ธาตุ counter ~100%, rarity +1 ~100% · upset ~1%
  → **PvE/หอคอย = โอเค** (โตทีละชั้น + ธาตุเป็นพัซเซิล) · **PvP = เสี่ยง pay/grind-to-win → รางวัล cosmetic/ต่ำ หรือจับคู่ตามพลัง**
- หมุดจูนแล้ว: elementAdv **1.25/0.85** (เดิม 1.5/0.67 ชนะขาด), crit 12%×1.6, variance ±22%, grade step ~+6-11%
- "element of surprise" = crit + variance ในแมตช์สูสี + ไม่รู้ทีมศัตรู + passive proc (ไม่ใช่สุ่มพลิกทีมพลังต่าง = จะรู้สึกดวงล้วน)

### 🎯 Passive (วัดด้วย `scripts/battle-passive-test.mjs`)
- **กฎทอง: passive = signature เฉพาะ epic/legendary (common/rare เปล่า)** → ทีมมี ~1-2 passive = บาลานซ์เอง
  - ยืนยัน: ทุกตัวมี passive → แรงเกิน 75-96% · 1 ใน 4 ตัว → พอดี 53-58% ✓
- **อย่าผูกพลัง passive กับ rarity แบบ stack** — rarity ให้ stat + "มีช่อง passive" ไม่ใช่ passive แรงกว่า
- เน้นแบบ **เงื่อนไข/sustain** (rage เลือดต่ำ, heal) — บาลานซ์ง่าย + เป็นไฮไลต์ให้ดู + ต้าน snowball
- หมุด archetype เข้าเป้า (~55-58%): shield ลดหมัดแรก 50% · dodge 10% · heal 8% ของดาเมจ · thorns 10% · rage +20% เมื่อ HP<40% · firstStrike (ตัวนำ 1 ตัว — ยัง 67% ต้อง nerf อีก)

### 🎬 ข้อจำกัดอนิเมชัน (emoji = ภาพนิ่ง)
- ทำได้แค่ขยับภาพนิ่งด้วย CSS: พุ่ง/สั่น/แฟลช/เด้งหาย/เลขลอย/จอสั่น/สาด ✨💥 — ไม่มีเฟรมท่าทาง
- brawl 8 ตัวบนจอแคบ → emoji ~44px, 2 แถว/ฝั่ง, หลอดเลือดรวมต่อทีม + จุดรายตัว
- หลาย action/รอบ → เล่นทีละ action เร็ว (~150-220ms) + สปอตไลต์ตัวที่ตี → รอบ ~5-8 วิ
- engine แยกจากอนิเมชัน → ทำ v1 ดิบ (เลขเด้ง+แฟลช) ก่อน ค่อยเติม juice

### เครื่องมือ + data
- **เครื่องมือจูน:** `scripts/battle-sim.mjs` (สมดุล) + `scripts/battle-passive-test.mjs` (passive) — แก้ CFG/PARAMS รันดู win% ก่อนลงโค้ด Vue
- **ต้องสร้าง:** `utils/battleEngine.js` (port จาก sim), `data/towerFloors.js`, signature passive ใน pet data
- **PvP async** (เฟสหลัง): snapshot ทีมศัตรูใน Firestore → resolve client-side · ⚠️ client-only โกงได้ → รางวัลต่ำ/cosmetic

## 5.5 ระบบ Passive (signature ต่อตัว)

### กฎดีไซน์
- **เน้นพลังมีเงื่อนไข/synergy** (build-around) > พลังแบนตลอดเวลา (ดู §5 ผลบาลานซ์ passive)

### Engine: 6 hook + aura ครอบกลไกทั้งหมด
`onBattleStart · onTurn · onAttack · onHit(โดนตี) · onKill · onDeath` + **team-aura** (คิดตอนจัดทีม, แก้ stat ก่อนสู้)
→ effect primitives ~20 แบบ (aoe-opener, on-kill-chain, scale-with-kills, regen, target-lowest, revive, guardian, dmg-reduction, cleave, multi-hit, heal-ally, shield-ally-death, execute, dodge, extra-hit, team-debuff-aura, thorns, atk-when-full, cheat-death, team-heal-opener, duo-synergy)
→ **เพิ่ม passive ใหม่ = เพิ่ม data ไม่แก้ engine**

### Data shape (เสนอ)
```js
pet.passive = {
  name: 'เกราะหนาม', icon: '🛡️',
  hook: 'onHit',            // จุด trigger
  effect: 'thorns',         // primitive id
  value: { pct: 10 },       // ตัวเลข — บาลานซ์ทีหลัง
  target: 'self',           // self | lowestAllyHp | allEnemies | ...
  filter: null,             // 'fist' | 'common' ... (สำหรับ aura)
  duo: null,                // 'seal' ถ้ามีคู่หู
}
```

### Banner-on-proc (UI) — ป้ายชื่อ "บนหัว pet" ตามจังหวะจริง
- engine ใส่ event `{type:'passive', petId, name, atRound}` ลงใน log ทุกครั้งที่ passive ทำงาน
- replay เจอ event → เด้งป้ายชื่อ passive **เหนือหัว pet ตัวนั้น** ~0.6 วิ (CSS pop+fade) — **ไม่ใช่กลางจอ**
- **จังหวะตามที่ proc จริง:**
  - proc ตอนเริ่มเกม → ป้ายของทุกตัวที่ trigger ขึ้น**พร้อมกัน**ตอนเริ่ม
  - pet เดียวมี passive proc 2 อัน → **เรียงขึ้นทีละอัน** (stack/คิว เหนือหัวตัวเดียวกัน)
  - proc ระหว่างสู้ → ขึ้นตอนนั้นเหนือหัวตัวที่ทำงาน

### Duo (คู่หู)
- whale 🐳 + seal 🦭 = คู่แรก (เข้าทีมด้วยกัน → บัฟพิเศษ) · กลไก `duo-synergy` เช็ก roster

- เปิดทางทำคู่อื่นอนาคต (เช่น มังกร+บาฮามุท สายมังกร)

## 5.6 🏆 Weekly Tournament (Phase 3 — สังคม)

ทัวร์ PvP รายสัปดาห์ · single-elimination · async (ใช้ battle engine เดิม) · สายเดียวรวม (แชมป์ชนะยาวได้ ไม่เป็นไร)

### ใครรัน bracket? (client-only ไม่มี backend)
- ลงทะเบียน → **snapshot ทีม** (4 ตัว + เกรด + passive) ล็อกใน Firestore (แก้ไม่ได้ระหว่างทัวร์)
- ถึงเดดไลน์ → **แอดมินกดปุ่ม resolve** → client รันทั้ง bracket ด้วย engine (deterministic + seed เดียว) → เขียนผล + battle log ลง Firestore
- ทุกคนแค่ **อ่านผล + ดู replay** · seed เดียว = ใครรันซ้ำได้ผลเดิม → ตรวจสอบได้/กันโกง
- *ทางเลือก:* Cloud Function รันอัตโนมัติ (สะอาดกว่า แต่หลุด no-backend)

### Bracket + bye
- ขนาด = 2^n ที่ ≥ จำนวนคน · บาย = (ขนาด − คน) แจกให้ซีด · ตัวอย่าง 14 คน → 16 (2 บาย) → 8 → 4 → 2 → 1
- **ซีด = สุ่ม** (แฟร์/ลุ้น) ไม่ซีดตามพลัง (กัน bye เอื้อตัวเต็ง)

### รางวัล (อยู่ในเพดานเศรษฐกิจสกุลเดียว — อย่าให้ทัวร์พิมพ์เหรียญจนเฟ้อ)
- **เข้าร่วม:** ทุกคนที่ลง → เหรียญ/ตั๋วกาชาคุ้มๆ (คนอ่อนก็อยากลง)
- **ลำดับ:** badge + เหรียญ (ไม่เยอะ) — แชมป์/รองแชมป์/รอบลึก ลดหลั่น
- **Cosmetic ชุดเริ่มต้น (reuse ระบบ tag/emoji เดิม):**
  - **Badge แชมป์ 🏆** ผ่านระบบ `tags.js` (auto-assign ตอน resolve) — ถาวร
  - **มงกุฎ 👑 ข้างชื่อ** เฉพาะแชมป์สัปดาห์ล่าสุด (หมุนเวียน ใครเป็นแชมป์ใหม่ก็ย้าย)
  - **Hall of Fame** หน้ารวมแชมป์ทุกสัปดาห์ (ถาวร น่าภูมิใจ)
  - *(cosmetic เว็บทำได้ = CSS+emoji: สี/เรืองแสง/กรอบ/มงกุฎ/ถ้วย/ป้าย — ทำสกินวาดเองไม่ได้)*

### Cadence + replay
- **ลงทะเบียน (อา) → ล็อกทีม → admin resolve (ภายใน จ) → ดูผล+replay ทั้งสัปดาห์ → รับรางวัล** · ใช้ scheduled task แจ้งเตือน
- **Replay:** เก็บ seed + ทีมของแต่ละแมตช์ → ดูย้อนได้ด้วย battle UI เดิม (โดยเฉพาะไฟนอล = สเปกเตเตอร์)
- **UI บราคเก็ต (มือถือ):** จอ ~380px → ผังเต็ม 16-32 ช่องอัดเกิน · ใช้ **scroll แนวนอน** หรือ **แท็บทีละรอบ** (16→8→4→ชิง) อ่านง่ายกว่า · ชิป = ชื่อ + emoji 4 เพ็ท + ผลแพ้ชนะ · แตะคู่ → replay · จอใหญ่ค่อยโชว์ผังเต็ม 2 ฝั่งแบบคลาสสิก

### Data model
`tournaments/{weekId}` `{ status: registration|locked|resolved, bracketSize, seed, registrants:[{uid,name,teamSnapshot}], bracket:[round][match], results, rewardsClaimed }`

### Rules / กันโกง (client-only = trust-based)
- registration = create-only โดยตัวเอง · ตรวจ snapshot ว่าเป็นเพ็ทที่ owns จริง (light validate)
- bracket/results = admin-write เท่านั้น · deterministic = ทุกคน verify ได้ → รางวัลต่ำ/cosmetic ลดแรงจูงใจโกง

> หมายเหตุ: ทัวร์ = PvP → snowball (ทีมแกร่งกว่าชนะ) เป็นเรื่องปกติ คุมด้วย "รางวัลเข้าร่วมเป็นหลัก + ลำดับเน้น cosmetic/เกียรติ"

## 5.7 👹 World Boss (Phase 3 — co-op รายสัปดาห์)

บอสรุ่น — ทั้งชั้นปีรุมตัวเดียว · เน้น **ร่วมมือ** ไม่ใช่แข่งกัน · reuse battle engine (ทีมเรา vs บอส)

### กลไกหลัก
- **บอส = damage race** — HP ก้อนใหญ่ (เกินทีมเดียวจะฆ่าได้) · แต่ละ attempt = "ดาเมจที่ทีมเราออกได้ในศึกนั้น" (engine เดิม แต่**นับ damage** แทนแพ้/ชนะ) → สะสมลด HP รวมของบอส
- **attempt จำกัด 3 ครั้ง/คน/บอส** · ซื้อเพิ่มด้วยตั๋ว (cap เช่น +3 กันคนเดียว solo) · ตั๋วจาก expedition/quest/shop = sink
- **HP bar live** (`onSnapshot`) — ทั้งรุ่นเห็นเลือดบอสลดเรียลไทม์ = ลุ้นร่วมกัน · leaderboard ดาเมจ (เกียรติ/cosmetic)

### รางวัล (2 ชั้น — กัน whale-dominance)
- **ล้มบอสได้ก่อนเดดไลน์ → ทุกคนในรุ่นได้ base reward** 🎉 (สร้างความเป็นรุ่น)
- **คนที่ร่วมสู้ / ดาเมจสูง → extra** (กันคนไม่เล่นได้ฟรีเฉยๆ + ให้กำลังใจคนช่วย) · ทั้งหมดอยู่ในเพดานเศรษฐกิจ

### Flow + กันโกง
- **admin กด spawn (พฤ)** → เปิดให้ตี → **admin/scheduled ปิด (อา) → แจกรางวัล**
- client-only: client คำนวณดาเมจเอง → `increment(-dmg)` atomic บน boss HP · cheatable → รางวัลเน้นร่วมมือ/ต่ำ ลดแรงจูงใจ + admin ตรวจยอดตอนปิดได้
- *(บอส battle ต่างจาก PvP: ไม่มี "ตายเกลี้ยงชนะ" → จบเมื่อครบ N รอบ หรือทีมเราหมด แล้วนับดาเมจสะสมที่ลงบอส)*

### ปฏิทินรายสัปดาห์ (2 event คนละครึ่งสัปดาห์ — ไม่ชนกัน)
```
อา : ลงทะเบียนทัวร์
จ  : admin resolve ทัวร์ → แจกรางวัล + ดู replay
พฤ : admin spawn World Boss → ทั้งรุ่นรุมตี
อา : ปิดบอส → ถ้าล้มได้ แจกรางวัลทั้งรุ่น
```
> ⚠️ 2 event/สัปดาห์ พ่นรางวัลทั้งคู่ → **คุมรวมให้อยู่ในเพดานเศรษฐกิจ** (อย่าให้ tournament+boss+expedition รวมกันเฟ้อ)

## 5.8 🗺️ Expedition (ส่งเพ็ทผจญภัย — idle/collection)

ส่งเพ็ทที่ไม่ได้ลงทีมไป "ผจญภัย" แบบจับเวลา → กลับมาได้รางวัล · **แก้ปัญหา "มี 27 ตัวแต่ใช้แค่ 4"** → ทุกตัวมีงาน = ให้คุณค่าการสะสม/กาชา

### กลไก
- **มิชชันจับเวลา** (เช่น 1 / 4 / 8 ชม.) · ส่งเพ็ทไป → ครบเวลากดเก็บ → ลูปเช็กอินรายวัน (ดึงคนกลับมาทุกวัน)
- **slot ปลดตามเลเวลบ้าน** (reuse residence ladder) — บ้านสูง = ส่งได้หลายสาย
- **มิชชันมีธาตุ** — ด่านนี้อยากได้สาย fist → ส่งสาย fist ได้รางวัลดีกว่า → เอาธาตุมาใช้นอกการสู้ (กลยุทธ์เบาๆ)
- (ออปชัน) เพ็ทที่กำลังผจญภัย = busy ลงทัวร์/บอสไม่ได้ → สร้าง tradeoff "รายได้ vs ใช้สู้" · หรือคงง่ายๆ ใช้ได้ปกติ

### รางวัล = **ของที่ไม่ใช่เหรียญ** (สำคัญ — กันเฟ้อ + ป้อนระบบอื่น)
- ตั๋วกาชา · **ตั๋ว attempt บอส** · shard อัปเกรดเพ็ท · token ทัวร์ · ของตกแต่ง
- → ไม่ inflate สกุลเงิน + เป็น "ก๊อกของรางวัลแนวใหม่" ที่เชื่อมเข้า gacha/boss/tournament

### Data + ความง่าย
- `user.expeditions: [{ petIds, missionId, endsAt, claimed }]` · `data/expeditions.js` (มิชชัน + ตารางรางวัล)
- ทำง่ายมาก (timer + reward table ไม่มี combat ใหม่) → เป็นระบบ "คุ้มสุด" ในกลุ่ม social/idle

## 6. ลำดับการสร้าง (build order)

```
1. เศรษฐกิจสกุลเดียว: cap ฟาร์ม + daily quest ปลดตัวคูณ ×1.25   ← ทำก่อน (ฐาน)
2. Pet roster rework (27 ตัว) + กาชา rework: dupe→+เกรด + pity   ← ต้องมี cap ก่อน
3. Battle engine v1 + passive (signature) + หอคอย (PvE)         ← payoff ของเพ็ท (rarity+เกรด+ธาตุ)
4. Expedition (idle/collection — ทำง่าย ป้อนตั๋วระบบอื่น)         ← ให้คุณค่าการสะสมก่อน social
5. Weekly Tournament + World Boss (co-op) + cosmetic/badge      ← social event (สลับสัปดาห์)
6. Potential affix (เปิดตาม engine) + ตลาด/market               ← เลเยอร์ลึก endgame
```

> หลักตลอดทาง: **ความลึกของเพ็ทต้องมี payoff รองรับเสมอ** — อย่าปล่อย stat/affix ที่ยังไม่มีระบบใช้

---

## 7. วิธีจูนตัวเลข (ทำก่อนแตะเลขจริง)

1. เคาะ **หอคอยกี่ชั้น (v1=50)** + **เป้าไปถึง/เทอม (~35)**
2. เคาะ **ขนาดทีม (4)** + **เกรดที่ผ่านชั้นเป้าได้ (~5)**
3. ถอยกลับคำนวณ: free pull รายวัน + pull ที่ซื้อไหว → dupes/เทอม → ถึงเกรด 5 บนทีมหลักไหม → ปรับ copies/pity
4. ตั้ง stat บอทแต่ละชั้นให้ล้อกับเส้นพลังผู้เล่น
5. ตรวจ faucet รวม (บ้าน+ฟาร์ม+หอคอย+เรียน) ต่อวัน ≤ เพดานที่ตั้ง → กาชายังลุ้น

## 8. ร้านค้า + ระบบสนับสนุน (⏸ parked — ละเอียดอ่อน รอคุย)

> ยังไม่ลงรายละเอียด — user ขอเก็บไว้คุยทีหลังเพราะเป็นประเด็นอ่อนไหว

ปมที่ต้องระวังตอนหยิบมาคุย (จดไว้กันลืม):
- **pay-to-win / ความเป็นธรรมในชั้นปี** — มี tag `supporter` ที่ให้ `incomeBonusPct 20` อยู่แล้ว ถ้าผูกกับเงินจริง = คนจ่ายได้เปรียบ ต้องคิดเรื่องความรู้สึกเพื่อนร่วมรุ่น
- **เงินจริงในโปรเจค public** (GitHub Pages, ทั้งรุ่นใช้) — มีนัยเรื่องความเหมาะสม/กฎ
- ทางที่มัก "ปลอดภัย" กว่า: ของสนับสนุนเป็น **cosmetic-only** (ไม่กระทบพลัง/รายได้) เพื่อเลี่ยง pay-to-win — แต่ขอเก็บไว้ตัดสินใจตอนคุยจริง
- ร้านค้า (coin shop) ในเกม = ส่วนหนึ่งของ **sink** (ข้อ 1) ออกแบบคู่กับเศรษฐกิจ
